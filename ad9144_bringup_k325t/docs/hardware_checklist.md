# K325T AD9144 Bring-Up Checklist

Goal: prove the FMCADDA-9250-9144 digital link is alive before judging analog waveform quality.

## Required hardware

- K325T board.
- FMCADDA-9250-9144 daughter card installed on the FMC HPC connector.
- Vivado 2024.1 Enterprise with valid K325T/JESD license.
- Oscilloscope on the AD9144 analog output connector after digital link checks pass.

## Build output

Expected bitstream path:

```text
D:\FPGA\ad9144_bringup_k325t\vivado\ad9144_bringup_k325t.runs\impl_1\top.bit
```

Build command:

```powershell
& D:\vivado\Vivado\2024.1\bin\vivado.bat -mode batch -source D:\FPGA\ad9144_bringup_k325t\scripts\build_bitstream.tcl
```

If the batch shell hits `CScript Error: Loading your settings failed`, use the direct in-process build instead:

```powershell
& D:\vivado\Vivado\2024.1\bin\vivado.bat -mode batch -source D:\FPGA\ad9144_bringup_k325t\scripts\build_direct.tcl
```

The vendor ROM waveform comes from:

```text
D:\FPGA\ad9144_bringup_k325t\ip_data\sine.coe
```

`create_project.tcl` copies it to the old paths recorded inside the vendor 2018.3 `blk_mem_gen_0` IP before IP import/upgrade.

## Power-up checks before programming

- Confirm the FMC daughter card is physically seated.
- Confirm the K325T board powers normally.
- Confirm the FMCADDA card PLL indicator can turn green after the design configures LMK04828. The child-card manual says this green PLL LED indicates LMK04828 PLL lock.

## Vivado Hardware Manager checks

After programming `top.bit`, Vivado should report debug cores because the vendor demo instantiates ILA/VIO:

- `vio_for_jesd_rst`
- `my_ila_jesd` for TX
- `my_ila_jesd` for RX
- `ila_for_adc_data`

If Vivado only says "no supported debug core", then the wrong bitstream was programmed or the build removed ILA.

## First status signals to inspect

Use the TX ILA first. Probe mapping from vendor `top.v`:

| ILA probe | Expected after link-up | Meaning |
|---|---:|---|
| `probe0[0]` | `1` | GTX QPLL lock |
| `probe1` | `1` | JESD TX reset done |
| `probe2` | `1` | JESD TX accepts sample data |
| `probe3` | active / mostly asserted per JESD state | AD9144 SYNC status into FPGA |
| `probe4` | periodic pulses if SYSREF present | SYSREF sampled |
| `probe5` | periodic frame markers | TX start-of-frame |
| `probe7` | periodic multiframe markers | TX start-of-multiframe |
| `probe10/11` | changing sample data | waveform samples feeding JESD TX |

Then inspect the RX ILA only if ADC loopback/debug is needed:

| ILA probe | Expected after link-up | Meaning |
|---|---:|---|
| `probe1` | `1` | JESD RX reset done |
| `probe2` | `1` when ADC is streaming | RX data valid |
| `probe3` | active | AD9250 SYNC generated by FPGA |
| `probe9` | `0` | RX frame error |

## Scope check order

1. Do not start with the oscilloscope. First confirm LMK PLL lock and TX ILA status.
2. Once `QPLL lock = 1`, `TX reset done = 1`, and `TX tready = 1`, probe the AD9144 output.
3. The vendor ROM pattern is a repeated waveform from `blk_mem_gen_0`; do not expect the final AWG DDS behavior yet.
4. If the output is flat, check `i_tx_sync`, SYSREF, and AD9144 SPI init sequence before changing waveform code.

## Known risk items

- The vendor demo is originally a ported reference design, not a clean K325T-native project.
- The K325T-specific XDC in this folder was derived from the K325T base schematic sheet 15 and the daughter-card FMC signal table.
- `D:\awg_fpga\rtl\top\awg_fmc_adda_top.v` is currently only a skeleton and does not connect the JESD204 PHY. Use this bring-up project first.
